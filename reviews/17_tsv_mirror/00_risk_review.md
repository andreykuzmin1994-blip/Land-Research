## 1. Risk list

**R-M1 — HIGH — `except Exception` cannot deliver the "never fails the loop" promise: the DSN path calls `sys.exit`.**
`prepare._get_connection_dsn()` (prepare.py L241–250) calls `sys.exit(EXIT_DATABASE_URL_MISSING)` when `DATABASE_URL` is unset after `load_dotenv`. `SystemExit` derives from `BaseException`, so a `_mirror_log_row` written as "on ANY exception … returns False" via `except Exception` will not catch it: a missing/mislaid `.env` mid-run kills the loop at the first TSV append, kills the halt path, and kills the test process in any offline environment. This is the exact opposite of the design's core contract. Note the same latent hazard exists in `_check_db_connection` (runner.py L504–516) — precedent, not permission.
*Mitigation*: add a non-exiting DSN probe in `prepare` (e.g., `prepare.dsn_available() -> bool`, in-scope since this is already a prepare-mutation commit) and short-circuit `_mirror_log_row` to `False` before touching `get_connection`; additionally catch `SystemExit` explicitly as defense in depth. A test must simulate a `sys.exit`-ing connection factory and assert `False` is returned.

**R-M2 — HIGH — Offline test hermeticity and suite speed (SR-6).**
The loop/baseline tests (e.g., `TestPhase10ExperimentLoopBaselineBootstrap` at tests/test_discovery.py L5241–5301, and the clusters at L5310–5412, L6376–6446, L6567–6641, L6666–6790) patch `runner.verify_setup`, `runner.evaluate`, and `runner._git_head_commit` — **nothing patches `prepare.get_connection`**. Adding mirror calls at the append sites means every one of these tests either (a) hits R-M1's `sys.exit` (CI has no `.env`), or (b) in a dev container with `.env`, performs a **real network connect per appended row** (10 s `connect_timeout` each, and worse: real INSERTs into the live mirror from tests). The offline suite is advertised at ~1 s and runs unconditionally on every push (`offline-tests.yml`, reviews/16 decision note).
*Mitigation*: `_mirror_log_row` is the single choke point — design that seam deliberately: (1) the R-M1 fast-path makes CI green by construction; (2) loop-test classes patch `runner._mirror_log_row` in `setUp` (mirroring the `verify_setup` pattern) so dev machines with `.env` never write to prod from tests; (3) one dedicated test class exercises `_mirror_log_row` itself against `FakeConnection` via `mock.patch("runner.prepare.get_connection")` (same object as the `research.prepare` patch precedent at tests/test_discovery.py L125–135). Gate G2 below makes this binary.

**R-M3 — MED — Unbounded hang window: connect is bounded, the INSERT is not.**
`get_connection` sets `connect_timeout=10` (prepare.py L264) but no `statement_timeout`. A network partition after connect establishment can hang the INSERT for TCP-keepalive-scale durations **inside the loop between iterations** — the NEVER STOP rule violated by a hang rather than a stop, and time attributed to nothing (the TSV `wall_clock_min` is computed inside `evaluate`, so the ledger won't even show it). Worst case is the halt path: the L1129 caller fires precisely *because* the DB is dead (3× `verify_setup` failure), then `_record_halt_row`'s mirror attempt dials the dead DB again.
*Mitigation*: the mirror's private connection must set a server-side statement timeout (e.g., `options='-c statement_timeout=5000'` or equivalent psycopg3 kwarg) so total worst-case added latency per row is bounded (~15 s). Accept the bounded halt-path delay; document it.

**R-M4 — MED — Naming the table `experiment_log` is a canonicality hazard. Rename to `experiment_log_mirror`.**
The repo contains 40+ references to "experiment_log" that mean *the TSV*, and the entire contract rests on "TSV is canonical." A Postgres table with the identical name invites: future reporting/CLI code reading the *table* as if canonical; grep/doc ambiguity in every future review; confusion with the adjacent `research_log` table (prepare.py L447). Today there is no mechanical leak path — the metric SQL is hard-pinned to `parcel_scores` (`_LATEST_SCORE_CTE*`, prepare.py L624–641) with no dynamic table enumeration — so this is a future-maintainer hazard, which is exactly what naming is for. `experiment_log_mirror` makes the subordinate role self-documenting at zero cost. (Keep the column `commit_hash`, not `commit` — reserved-word adjacent — as designed.)

**R-M5 — MED — Purge-boundary confusion (SR-13): the mirror must be deletion-free, stronger than `parcel_scores`.**
`_SQL_DELETE_SCORES_FOR_EXPERIMENT = "DELETE FROM parcel_scores WHERE experiment_id = %s"` (runner.py L823–824). The mirror table also carries `experiment_id` — a future "revert everything for experiment X" sweep is a plausible one-line mistake that would destroy exactly the history the mirror exists to preserve. **Affirmed: discarded/crash/timeout experiments' mirror rows MUST persist** — the TSV records failures by design (that is the Karpathy log's entire value: `discard` rows are the learning), and the mirror mirrors the TSV, not the data ratchet. SR-13's "the ONLY sanctioned deletion" sentence must be updated in the same change to name the mirror as append-only with **no** sanctioned deletion at all.
*Mitigation*: SR-13 text update (design D) + a static AST/grep test asserting no `DELETE`/`UPDATE`/`TRUNCATE` against the mirror table exists in any pipeline module, alongside the existing `TestExperimentPurge`.

**R-M6 — MED — Live-mirror rows must be byte-identical to what the TSV holds: insert the `_validate_log_row` OUTPUT, not the input row.**
`_validate_log_row` (runner.py L273–333) is where sanitization and canonical stringification happen (`confidence` → `.2f`, `wall_clock_min` → `.1f`, description whitespace-collapsed and truncated at 200 with `…`). If `_mirror_log_row` inserts the caller's *raw* dict, the mirrored `description` can differ from the TSV line (unsanitized whitespace), which (a) breaks the mirror-of-record property and (b) breaks backfill dedup equality (`description` is a key column), producing duplicates on every backfill. The design says "validates via `_validate_log_row`" — make it explicit that the *validated dict* is what gets inserted (cast `metric`/`api_calls` back to int and `confidence`/`wall_clock_min` via the formatted strings; psycopg3's unknown-type text params do resolve against INTEGER/NUMERIC, but don't rely on that silently).
*Mitigation*: property-style test — for a given row, the tuple `_mirror_log_row` executes equals the parsed TSV line `append_experiment_log_row` wrote.

**R-M7 — MED — Backfill dedup: real false-positives exist, and the EXISTS-then-INSERT pattern races (SR-9, SR-5).**
The key `(commit_hash, metric, status, description)` is not unique over legitimate TSV content: outer-crash rows carry `experiment_id=None` (runner.py L1149–1159) so their descriptions have **no** `exp=` token — two identical consecutive outer crashes at the same commit are byte-identical rows, and the second will never backfill (false positive). Same for repeated halt rows (`"halt | market=X | halt sentinel detected"`, commit possibly `pending`). Conversely a UNIQUE index is impossible for the same reason, so two concurrent backfills double-insert (false negative).
*Mitigation*: count-based reconciliation (insert `tsv_count − mirror_count` copies per key, in TSV order) fixes the false positive cleanly; wrap the backfill in `pg_advisory_xact_lock` to kill the race; document "operator-invoked, one at a time" regardless. If count-based is judged too much machinery, the false positive must be explicitly accepted in the decision note — it is a durability gap, not a metric gap.

**R-M8 — MED — Marker parsing out of descriptions is fragile in three specific ways.**
(a) Crash-row descriptions embed the error string *before* the trailing `run=`/`exp=` tokens (`_format_loop_description`, runner.py L1272–1299); an error message containing `run=` or `exp=` would be matched first by `re.search` semantics (`_DESCRIPTION_RUN_RE`, L463, takes the FIRST match). Backfill must take the LAST match. (b) `_sanitize_description` truncates at 200 chars (L268–269) — a long description amputates or *truncates mid-id* the trailing `exp=` token; a naive parse then stores a wrong experiment_id. (c) Pre-F1/F4 legacy rows have no markers at all.
*Mitigation*: last-match parsing; validate parsed ids against the actual shape `^exp-\d{8}T\d{6}Z-[0-9a-f]{6}$` (from `_make_experiment_id`, runner.py L662–669) and run tags against `[a-z0-9._-]+` (reuse the existing pattern) — on mismatch store NULL; document that backfill-parsed `run_tag`/`experiment_id` are best-effort annotations, never authoritative (the live-threaded values are).

**R-M9 — MED — CI auto-applies this DDL to the live Supabase project on push, and `IF NOT EXISTS` cannot heal a wrong first schema.**
`validate-phase1.yml` re-runs `python prepare.py` against prod on any push touching `prepare.py`. The mirror table will exist in prod the moment the branch is pushed — before review completes. If a column type is wrong on the first push, `CREATE TABLE IF NOT EXISTS` silently no-ops forever after; fixing it requires an explicit ALTER (precedent: `_DDL_PARCEL_SCORES_RUN_COLUMNS`) or a manual DROP.
*Mitigation*: schema is frozen by review BEFORE first push (gate G6); types checked against `_validate_log_row` output (metric non-negative int → INTEGER fine; confidence/wall_clock NUMERIC fine; commit_hash TEXT must admit `"pending"` per `_TSV_COMMIT_RE` — no hex CHECK; `run_tag`/`experiment_id` nullable — required for halt/breaker/outer-crash and off-branch baseline rows). Any post-push schema change is a declared ALTER migration, not an edited CREATE.

**R-M10 — MED — The design has no restore path. Durability without documented restoration is half a feature.**
Backfill is TSV→DB only. The motivating scenario is container reclaim destroying the TSV — at which point the operator must reconstruct `experiment_log.tsv` from the mirror by hand, and every consumer (`_row_run_tag` anchor selection, `_last_baseline_or_keep`, `cli.py log/tail`, cross-run aggregation) reads the TSV. A hand-rolled restore (wrong column order, missing header, re-sanitized descriptions) would silently corrupt anchor selection for the next run.
*Mitigation*: at minimum, document the exact restore query + column order in STORAGE_ARCHITECTURE.md next to the table (ORDER BY entry_id, emit the 7 TSV columns verbatim); ideally a `make mirror-restore` sibling writing via the existing validated writer. May be deferred, but the decision note must say so explicitly — otherwise the durability claim is overstated.

**R-M11 — MED — SR-16 adjacency: after a reclaim, the mirror IS the record — tampering stops being cosmetic.**
`research.py` runs in-process with full DSN privileges and its own SQL constants; an adversarial sandbox could INSERT forged rows or DELETE mirror rows. While the TSV lives, this is harmless to decisions (the keep-or-revert anchor reads the TSV only) and detectable by divergence. **After container reclaim, the forged mirror is all that survives, with nothing to reconcile against.** This is a real upgrade of the accepted SR-16 surface, in the disaster case the mirror exists for.
*Mitigation*: one-line SR-16 update naming the mirror; make `make mirror-backfill`'s summary counts double as a divergence canary (already-present vs inserted vs mirror-only counts) and recommend running it routinely while the TSV is alive; optional hardening if a separate DB role ever exists: REVOKE UPDATE/DELETE on the mirror from the app role. Accepted-risk lineage otherwise.

**R-M12 — LOW — Metric-integrity (SR-15): clean today, needs a fence for tomorrow.**
Verified: neither `calculate_actionable_pipeline_count` nor `calculate_confidence_weighted_pipeline` (prepare.py L706–741) nor their CTEs reference anything but `parcel_scores`; the keep-or-revert anchor is TSV-only (`_last_baseline_or_keep`, runner.py L482–498). The mirror cannot leak into the metric without someone editing prepare's SQL — which is its own protocol. Residual risk is a future convenience ("read prior metric from the mirror instead of the TSV") quietly rerouting the decision input to a table the sandbox can write.
*Mitigation*: the only sanctioned pipeline reads of the mirror are the backfill dedup SELECTs; assert via static test that no other pipeline code SELECTs from it, and state in the mechanics doc note that the mirror is never a decision input.

**R-M13 — LOW — Backfilled rows' `logged_at`/`entry_id` misrepresent historical order.**
`logged_at DEFAULT NOW()` stamps backfill time, not experiment time; any mirror added mid-history (this one) or re-backfilled after partial loss interleaves wrongly by timestamp.
*Mitigation*: backfill in TSV order (entry_id then preserves relative order within a batch); add a `source TEXT NOT NULL DEFAULT 'live'` (`'live'`/`'backfill'`) column — one column that kills a whole class of future forensic confusion; document that TSV row order is the authoritative ordering.

**R-M14 — LOW — No CHECK on `status` is the right call; say why, and fix the adjacent doc drift.**
A CHECK would couple runner vocabulary (`_TSV_STATUSES`, runner.py L99–101) to prepare.py DDL — future vocabulary additions would then require a prepare-mutation, and a CHECK violation under the warn-only mirror contract silently stops mirroring the new status (invisible durability gap). `_validate_log_row` is the enforcement point. Note the pre-existing drift while here: mechanics L354 lists statuses without `halt`, which the code writes (SR-14 pattern) — the design-D doc pass should fix that line.

**R-M15 — LOW — Call-site placement and halt-row asymmetries.**
(a) Mirroring at the 4 call sites (rather than inside `append_experiment_log_row`) is correct — it keeps direct-append unit tests and `tests/test_cli.py` untouched and threads `run_tag` explicitly per the F2 precedent — but a future 5th append site can forget the mirror; backfill heals it, and a test pinning "every runner append site is paired with a mirror call" cheaply prevents it. (b) `_record_halt_row` gaining `run_tag` means live halt-row mirror rows carry a `run_tag` the TSV description lacks (halt descriptions have no `run=` token, runner.py L1256) — backfilled halt rows will have NULL there. Acceptable; document the asymmetry rather than discovering it later. All four `_record_halt_row` callers must pass the tag or the columns are inconsistently NULL.

## 2. GO / NO-GO gates (all must be TRUE before implementation is accepted)

- **G1**: `_mirror_log_row` cannot raise *anything* (including `SystemExit` — R-M1) and cannot block unboundedly (connect_timeout + statement_timeout — R-M3). Proven by tests: sys.exit-ing connection factory → returns False; raising cursor → returns False; both emit the loud warning with masked DSN only (SR-8).
- **G2**: full offline suite passes with `DATABASE_URL` unset, no `.env`, zero network, and no material runtime regression (currently ~1 s); no test can write to a live mirror table (R-M2).
- **G3**: TSV append precedes the mirror call at all 4 sites — mirror ⊆ TSV invariant (mirror-first would let a process death invent history the canonical log never had). Asserted by test on ordering.
- **G4**: diff shows `_SQL_DELETE_SCORES_FOR_EXPERIMENT`, both metric functions, `_LATEST_SCORE_CTE*`, `get_parameters`/sentinel logic byte-unchanged; static test proves no DELETE/UPDATE/TRUNCATE on the mirror table anywhere in pipeline modules (R-M5); no pipeline SELECT of the mirror outside backfill dedup (R-M12).
- **G5**: table named `experiment_log_mirror` (R-M4), or the decision note records an explicit operator override with reasoning.
- **G6**: schema frozen by review before first push (CI applies DDL to prod on push — R-M9); types match `_validate_log_row` output; `run_tag`/`experiment_id` nullable; no status CHECK; DDL is `IF NOT EXISTS` + plain `CREATE INDEX` (never CONCURRENTLY — apply_schema's one-transaction property, prepare.py L512–514, holds).
- **G7**: live mirror inserts the validated/sanitized row exactly as written to the TSV, proven by a round-trip equality test (R-M6); backfill parses markers last-match with shape validation (R-M8).
- **G8**: commit lands on `main` per protocol step 3 (current work branch is not main), message reads `prepare-mutation: add experiment_log mirror DDL — metric definition, gates, scoping, and parameters UNCHANGED; TSV metric values remain comparable across this commit`, and the decision note repeats the comparability carve-out so future analysis doesn't mechanically apply mechanics §Mutation step 6's "not comparable" rule to this commit. Next run starts a fresh tag/branch/baseline regardless (automatic — none exist).
- **G9**: SR-13 and SR-16 text updates plus the mechanics "Why Untracked" note land in the SAME change, wording preserving "TSV is canonical; the mirror is never a decision input"; restore path documented or its deferral recorded (R-M10).

## 3. Standing risks

SR-2, SR-5, SR-6, SR-8, SR-9, SR-15 apply and are covered by gates above; SR-10 clean (psycopg already pinned); SR-16 applies as adjacency. **Text updates required in this change**: SR-13 (add the mirror table: append-only, no sanctioned deletion, purge-exempt) and SR-16 (mirror named as a tampering surface whose canary is TSV↔mirror reconciliation). SR-14-adjacent: fix mechanics L354's missing `halt` status while editing that doc.

## 4. Verdict

**PROCEED-WITH-CHANGES.** The architecture is sound — TSV-first, warn-only, best-effort, explicitly threaded tags, operator-invoked backfill — and the metric/decision layer is verifiably untouched. But the design as literally specified fails its own core promise on two counts that would ship real incidents: the "on ANY exception" contract is unimplementable via `except Exception` because the DSN path calls `sys.exit` (R-M1), and the new call sites break offline-test hermeticity as written (R-M2). Those two, plus the `experiment_log_mirror` rename (R-M4) and the frozen-schema-before-push discipline forced by CI's live DDL application (R-M9), are conditions of the GO. The missing restore path (R-M10) must be either added or explicitly deferred in the decision note — durability that can't be restored is a comfort blanket, not a backup.
